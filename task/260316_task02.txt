### 2. `hashIp()` — 보안상 부적절한 해싱
**파일:** `src/lib/utils.ts` (42~50행)

**문제:** IP 해싱에 단순 djb2 해시를 사용하고 있습니다. 이는 **암호학적 해시가 아니며**, 충돌이 빈번하고 역산이 용이합니다. IPv4 주소는 약 43억 개뿐이므로 무차별 대입으로 원본 IP를 쉽게 복원할 수 있습니다.

**지침서 Task 5-3:** "IP를 SHA-256 해시"

**현재 코드:**
```typescript
export function hashIp(ip: string): string {
  let hash = 0;
  for (let i = 0; i < ip.length; i++) {
    const char = ip.charCodeAt(i);
    hash = ((hash << 5) - hash) + char;
    hash = hash & hash;
  }
  return Math.abs(hash).toString(16).padStart(8, "0");
}
```

**수정 방향:**
```typescript
import { createHash } from "crypto";
export function hashIp(ip: string): string {
  return createHash("sha256").update(ip + process.env.IP_HASH_SALT).digest("hex");
}
```

### 3. `cn()` 유틸 — clsx를 import만 하고 미사용 (기능 결함)
**파일:** `src/lib/utils.ts` (1~4행)

**문제:** `clsx`의 `ClassValue` 타입을 import하지만 실제 `clsx()` 함수를 사용하지 않고, 단순 `filter(Boolean).join(" ")`으로 대체합니다. 이는 clsx의 핵심 기능(조건부 클래스, 객체/배열 지원)이 동작하지 않습니다.

```typescript
import { type ClassValue } from "clsx"; // 타입만 가져옴
export function cn(...inputs: ClassValue[]) {
  return inputs.filter(Boolean).join(" "); // clsx 미사용
}
```

**수정 방향:**
```typescript
import { clsx, type ClassValue } from "clsx";
import { twMerge } from "tailwind-merge";
export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs));
}
```

### 4. `generateOrderNumber()` — 중복 가능성
**파일:** `src/lib/utils.ts` (7~11행)

**문제:** `Math.random()`으로 4자리 랜덤 숫자를 생성하여 주문번호를 만듭니다. 같은 날 9000건 이상의 주문이 아니어도 **생일 문제**에 의해 ~50건만으로도 약 1% 충돌 확률이 발생합니다.

```typescript
const rand = Math.floor(Math.random() * 9000) + 1000;
return `CF-${date}-${rand}`;
```

**수정 방향:** DB의 auto-increment 또는 해당 날짜의 기존 주문 수를 세어 순차 채번해야 합니다.
```typescript
// DB에서 해당 날짜 마지막 번호 조회 후 +1
const count = await prisma.order.count({
  where: { orderNumber: { startsWith: `CF-${date}` } }
});
return `CF-${date}-${String(count + 1).padStart(4, "0")}`;
```

### 8. `requireUser()` 패턴 — 에러를 catch로 삼킴
**모든 API 라우트** (`quotes/route.ts`, `pricing/route.ts`, `intake/route.ts` 등)

**문제:** 모든 인증 API에서 동일한 안티패턴이 사용됩니다:
```typescript
const user = await requireUser().catch(() => null);
if (!user) return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
```

`requireUser()`가 예외를 던지도록 설계되었는데 이를 catch로 삼키면, 인증 실패와 서버 오류(DB 연결 실패 등)를 구분할 수 없습니다. DB 오류도 401로 응답하게 됩니다.

**수정 방향:**
```typescript
try {
  const user = await requireUser();
  // ... 로직
} catch (err) {
  if (err instanceof AuthError) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }
  return NextResponse.json({ error: "Internal Server Error" }, { status: 500 });
}
```

### 9. API 경로 불일치 — `projects` vs `pages`
**지침서 섹션 3 프로젝트 구조:**
```
/api/pages/[id]/draft/route.ts
/api/pages/[id]/preview/route.ts
/api/pages/[id]/publish/route.ts
```

**실제 코드 (builder 페이지 87행, 118행, 127행):**
```typescript
fetch(`/api/pages/${pageId}/draft`, { method: "PATCH" });
fetch(`/api/pages/${pageId}/preview`, { method: "POST" });
fetch(`/api/pages/${pageId}/publish`, { method: "POST" });
```

**문제:** 빌더는 `/api/pages/[id]/...`를 호출하지만, 업로드된 소스에는 이 라우트 파일들이 **존재하지 않습니다**. 완료 보고서에는 `/api/projects/[id]/autosave`, `/api/projects/[id]/preview` 등으로 구현했다고 되어 있어 **경로가 불일치**합니다.

**결론:** 빌더의 핵심 저장/미리보기/배포 기능이 동작하지 않을 가능성이 높습니다.

**파일:** 모든 admin 페이지

```typescript
const projects = await prisma.project.findMany({ take: 100 });
```

페이지네이션이 구현되지 않아, 데이터가 100건을 초과하면 나머지를 볼 수 없습니다.

### 32. 빌더 — 테마/accentColor 변경이 서버에 저장되지 않음

테마와 Accent Color를 변경해도 `draftContent`만 자동 저장되고, Page의 `theme`과 `accentColor` 필드는 업데이트되지 않습니다.

---

### 33. 빌더 — "버전 저장" 기능 미구현
**지침서 Task 5-1:** '"버전 저장" 버튼 → PageVersion 생성'

빌더 상단에 "저장", "미리보기 생성", "배포" 버튼만 있고 "버전 저장" 버튼이 없습니다.

### 34. 빌더 — 반응형 대응 미흡
**지침서 Task 3-1:** "하단 탭바 (모바일 767px 이하, 핵심 5개만)"

빌더의 3패널 레이아웃이 모바일에서 사용 불가능합니다. 좌측 208px + 우측 288px = 최소 496px + 중앙 프리뷰 영역이 필요합니다.

### 35. fetchNotifications — useEffect 경고
**파일:** `src/app/dashboard/notifications/page.tsx` (29행)

```typescript
useEffect(() => {
  fetchNotifications();
}, []);
```

`fetchNotifications`가 의존성 배열에 포함되지 않아 ESLint 경고가 발생할 수 있습니다. `useCallback`으로 감싸거나 의존성에 추가해야 합니다.

---

### 36. 공통 에러 처리 부재
모든 클라이언트 컴포넌트에서 API 호출 에러를 각자 다른 방식으로 처리합니다. 중앙화된 에러 핸들러나 toast 시스템이 없습니다.

---

### 37. Admin 공지 — Promise.all로 대량 알림 발송
**파일:** `src/app/api/admin/notify/route.ts` (31~33행)

```typescript
await Promise.all(
  users.map(u => sendNotification({ ... }))
);
```

사용자가 수백 명이면 동시에 수백 개의 DB/이메일 요청이 발생합니다. 배치 처리 또는 큐를 사용해야 합니다.
